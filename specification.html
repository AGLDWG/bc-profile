1. Introduction

What this is

This document defines a profile of cataloguing standards with the definition of 'profile' coming from the Profiles Vocabularyref and it is:

A specification that constrains, extends, combines, or provides guidance or explanation about the usage of other specifications.

Here, the other specification being profiled is actually 2: DCAT & schema.org.

What it does

In the next section, this document describes how DCAT and schema.org elements must be presented - in certain arrangements wIth respect to one another and with certain predicates to indicate properties - to make a data that conforms to this profile.

This specification's rules/requirements - are numbered and indicated in red text.

This specification defines requirements for the cataloguing of generic resources according to catalogue models ensuring minimum object metadata is provided in particular forms. This allows for basic catalogue operations, such as filtering and order of resources by agents related to them (e.g. publisher), resource coverage (spatial and temporal) and so on.

This specification depends on the use of Semantic Web methods. It mandates that a minimal set of several Semantic Web model elements be used.

The models are the W3C's Data Catalog Vocabulary, DCAT, and also & schema.org, the widely-used, general-purpose web object description vocabulary which also defines DCAT-like data cataloguing elements.

How it is used

Other, more specialised, profiles exist that mandate more specialised Semantic Web models' elements to describe particular kinds of catalogued objects, in particular:

  • Ontology Publication Profile, OntPub: to characterise Semantic Web ontologies
  • Vocabulary Publication Profile, VocPub: to characterise classification vocabularies

These more specialised profiles also mandate the same basic metadata mandated here.

This profile is used by multiple Australian government and international catalogues that contain other catalogues and is entirely generic, i.e. the additional constraints are not Australian or government-specific but just ensure certain technical model capabilities in the data.

Updates

This profile is always open to being updated to match modern cataloguing requirements, so if you would like to contribute to or challenge its requirements, please contact its developers via the Code Repository which contains an Issues tracker.

2. Requirements

The requirements of this profile are placed primarily on the DCAT's Resource class and schema.org's CreativeWork class which both represent generic things that can be catalogued. In DCAT's case, Resource instances are contained in Catalog isntances, in schema.org's CreativeWork instances are contained in DataCatalogue instances.

Apart from constraining predicates on Resource & CreativeWork class instances, this profile also mandates some predicates be present on objects referred to by them, in particular PROV Agent instances - people and organisations - to which Resource & CreativeWork instances are attributed. These must at least have an identifier, a name and be classed as an Agent. See the Agent section below.

2.1 Resource

Resources are instances of either DCAT's Resource class or schema.org's CreativeWork class.

§ 2.1.1 Each resource MUST be identified by a IRI

To ensure resources can be catalogued effectively and governed:

§ 2.1.2 Each resource MUST have exactly one name and at least one description indicated using the schema:name or dcterms:title predicate and the schema:description or dcterms:description predicate, respectively that must give textual literal values. Only one definition per language is allowed

§ 2.1.3 Each resource MUST have exactly one created date and exactly one modified date indicated using the schema:dateCreated and schema:dateModified or dcterms:created and dcterms:modified predicates respectively that must be either an xsd:date, xsd:dateTime or xsd:dateTimeStamp literal value

§ 2.1.4 Each resource MUST have at least one creator, indicated using schema:creator or dcterms:creator predicate and exactly one publisher, indicated using schema:publisher or dcterms:publisher, all of which MUST be IRIs indicating instances of schema:Person, or schema:Organization. A prov:qualifiedAttribution predicate indicating an Agent with the prov:hadRole predicate indicating the value role:originator or role:publisher may be used instead of schema:creator & schema:publisher, respectively

§ 2.1.5 The origins of a resource MUST be indicated by at least one of the following predicates: prov:wasDerivedFrom, schema:citation, dcterms:source or skos:historyNote. The first SHOULD only link to other Semantic Wbe objects, the second SHOULD indicate either a URL or citation text and the third and fourth SHOULD present text

§ 2.1.6 The status of the resource, within the catalogue it is presented, SHOULD be given with the predicate schema:status indicating a Concept from the Registry Item Status Codes vocabulary or an extension of it

§ 2.1.7 High-level theming of a resource SHOULD be given using the schema:keywords or the dcat:theme predicate indicating Concepts from a vocabulary. Text literal values for either predicate SHOULD NOT be used

§ 2.1.8 Any licence pertaining to the reuse of a resource's content SHOULD be given using the schema:license or dcterms:license predicate preferentially indicating the IRI of a license if in RDF form or a literal URL (datatype xsd:anyURI) if online but not in RDF form. If the licence is expressed in text, a literal text value may be given

§ 2.1.9 Any policies other than licenses pertaining to the resource SHOULD be given using the odrl:hasPolicy predicate preferentially indicating the IRI of a policy if in RDF form or a literal URL (datatype xsd:anyURI) if online but not in RDF form. If the policy is expressed in text, a literal text value may be given. If in RDF, the policy should be formulated according to ODRL

§ 2.1.10 The copyright holder for the resource SHOULD be given using the schema:copyrightHolder predicate preferentially indicating the IRI of an Agent or a Blank Node instance of an Agent containing details as per Agent requirements. A prov:qualifiedAttribution predicate indicating an Agent with the prov:hadRole predicate indicating the value role:rightsHolder may be used instead of schema:copyrightHolder

2.2 Agent

To be consistent with other Semantic Web representations of Agents, vocabularies' associated Agents, creator & publisher must be certain typed RDF values:

§ 2.2.1 Each Agent associated with a resource MUST be typed as an schema:Person or schema:Organization

To ensure human readability and association of Agents with their non-Semantic Web (real world) form:

§ 2.2.2 Each Agent MUST give exactly one name with the schema:name predicate indicating a literal text value

To ensure that Agents are linked to non-Semantic Web forms of identification:

§ 2.2.2 Each Agent MUST indicate either a schema:url (for organizations) or a schema:email (for people) predicate with a URL or email value
